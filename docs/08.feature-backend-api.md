# 08. 백엔드 REST API (Spring Boot)

## 개요

Spring Boot 3.5.14 + Java 17 + Gradle 기반 REST API 서버. 포트 8081에서 실행되며, React 프론트엔드(Vite 5173)의 요청을 처리한다.

---

## 아키텍처

```
Controller → Service → Repository → Entity
     │           │           │
  (HTTP)    (비즈니스 로직)   (JPA/DB)
```

- **Controller**: HTTP 요청/응답 처리, 파라미터 검증
- **Service**: 비즈니스 로직, 트랜잭션 관리
- **Repository**: JPA를 통한 DB 접근 추상화
- **Entity**: 테이블 매핑, Lombok으로 보일러플레이트 최소화

## 엔드포인트 목록

### 상품 API (`/api/products`)

| Method | Path | 파라미터 | 설명 |
|--------|------|---------|------|
| GET | `/api/products` | 없음 | 전체 상품 조회 |

**응답 예시:**
```json
[
    {
        "id": 1,
        "martId": 1,
        "name": "농심)멘토스레인보우",
        "price": 900,
        "eventType": "1+1",
        "imageUrl": "https://...jpg",
        "stock": 8
    }
]
```

### 매장 API (`/api/marts`)

| Method | Path | 파라미터 / Body | 설명 |
|--------|------|----------------|------|
| GET | `/api/marts/nearby` | `?lat=...&lng=...&radius=1` | 주변 CU 매장 검색 (거리순) |
| POST | `/api/marts` | `{name, address, latitude, longitude}` | 매장 찾기 또는 생성 |

**POST 요청 예시 (find-or-create):**

## Haversine 공식 (지리적 거리 계산)

MySQL Native Query로 구현된 거리 계산 쿼리:

```sql
SELECT m.*,
    (6371 * acos(cos(radians(:lat)) * cos(radians(m.latitude))
    * cos(radians(m.longitude) - radians(:lng))
    + sin(radians(:lat)) * sin(radians(m.latitude)))) AS distance
FROM mart m
[WHERE m.name = 'CU']  -- findNearbyCUs에서만 사용
HAVING distance <= :radius
ORDER BY distance
```

- 6371km = 지구 반지름
- `findNearbyCUs`: `WHERE m.name = 'CU'` 조건 포함 (기존 시드 매장용)
- `findNearby`: 이름 조건 없음 (동적 생성 매장 검색용)

## CORS 설정

```java
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
                .allowedOrigins("http://localhost:5173")
                .allowedMethods("GET", "POST", "PUT", "DELETE")
                .allowedHeaders("*");
    }
}
```

- `localhost:5173` (Vite dev 서버)의 API 호출만 허용
- 운영 환경에서는 도메인 변경 필요

## 문제 및 해결

### 문제 1: JPA N+1 문제 발생 가능

`MartService.findOrCreate()`에서 80개 상품을 `for` 루프로 개별 `save()` 호출. 이론적으로 N+1 문제가 있으나, `@Transactional` 내에서 실행되어 배치 처리가 이루어짐.

```java
// 개선 가능: batch insert로 최적화
for (CatalogProduct cp : catalog) {
    productRepository.save(product);  // 80번의 INSERT
}
```

→ 현재 규모(80개)에서는 성능 문제 없음. 상품이 수천 개라면 `saveAll()`로 개선 필요.

### 문제 2: DTO 없이 엔티티 직접 반환

보안/유지보수 측면에서 DTO 패턴이 권장되지만, MVP 단계에서는 단순성을 우선하여 엔티티를 JSON으로 직접 직렬화.

```java
// 현재: 엔티티 직접 반환
public List<Product> getAllProducts() { return productRepository.findAll(); }

// 개선 방향: DTO 분리
// public List<ProductResponse> getAllProducts() { ... }
```

### 문제 3: 입력값 검증 부재

`POST /api/marts`에서 `Map<String, Object>`로 요청 바디를 받아 타입 캐스팅. 잘못된 입력이 들어오면 `ClassCastException` 발생 가능.

```java
// 현재: raw Map 사용
public Mart findOrCreate(@RequestBody Map<String, Object> body) { ... }

// 개선 방향: @Valid + DTO 사용
// public Mart findOrCreate(@Valid @RequestBody MartCreateRequest request) { ... }
```

## 추가/변경 사항

- **초기**: Mart 엔티티에 `@Getter`만 있었으나, 동적 생성 기능 추가 시 `@Setter` 필요해짐
- **초기**: Product 엔티티에 `stock` 필드 없었으나, 재고 표시 기능 추가됨
- **MartRepository**: `findNearby()` 쿼리는 나중에 추가 (동적 매장 검색용)
- **MartService**: `findOrCreate()`는 최종 단계에 추가된 메서드
- No Spring Security, No validation, No DTO — MVP 단계의 최소 구성
